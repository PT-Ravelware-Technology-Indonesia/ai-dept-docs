## Project Overview
Sistem edge computing yang berjalan on premises untuk di gunakan di sorting Si-Cepat. Tugasnya adalah mengukur dimensi dan berat secara otomatis, lalu mengirimkan hasilnya ke backend melalui RestAPI (FastAPI) dan MQTT.

## Konteks Penggunaan
Sistem dipasang conveyor belt. dengan alur scan barcode → Timbang Berat → Hitung Dimensi → Kirim hasilnya melalui MQTT

# FITUR

## Pengukuran Dimensi Otomatis
Kamera depth Orbbec dipasang diatas platform timbangan menghadap kebawah. Setiap frame, sistem melakukan segmentasi depth untuk mendeteksi objek yang ada di bawah kamera. Jika objek menutupi atau bersinggunakn dengan titik tengah frame, sistemn akan mulai mengukur dimensinya dan merata-ratakan hasilnya dari beberapa frame sebelumnya.

- Menggunakan Hardware D2C (Depth to Color Alignment)
- Nilai Intrinsik kamera dobaca otomatis dari SDK saat Startup
- Floor depth dikalibrasi otomatis saat startup -- Bisa di konfigurasi menggunakan API /config
- **Edge-aware refinement**: piksel di tepi transisi depth (antara objek & lantai) yang depth-nya tidak reliabel dibuang menggunakan Sobel gradient, agar contour tidak melebar dan L/W tidak over-estimate. Dikontrol oleh `edge_gradient_threshold` (mm/px, 0 = nonaktif).

## Kamera RGB Sekunder (Foto Bukti Paket)
Selain kamera depth Orbbec, sistem dapat menggunakan **kamera USB webcam tambahan** sebagai sumber foto bukti paket yang lebih jelas. Kamera ini hanya melakukan capture (mengikuti trigger transaksi), bukan untuk pengukuran.

- Berjalan di background thread agar frame terbaru selalu siap saat trigger (tanpa lag warmup).
- **Non-fatal**: jika kamera webcam gagal/ tidak terpasang, sistem tetap jalan dan capture sekunder dilewati.
- Foto hasil webcam (jika tersedia) yang dikirim sebagai `image_base64` di payload `DATA/RESULT` dan disimpan ke folder `captured_images_rgb/`.
- Dapat dinonaktifkan via `RGB_CAMERA_ENABLED=False` di `.env`.

:::info{title="Deteksi Index Kamera"}

Index webcam diatur via `RGB_CAMERA_INDEX`. Jika index salah (mis. ter-claim oleh driver lain seperti RGB internal Orbbec), kamera bisa "terbuka" tapi tidak mengirim frame — sistem mendeteksi ini dalam 3 detik dan melewati capture sekunder. Gunakan `tools/list_cameras.py` untuk mengecek index yang benar.

:::

## Integrasi Timbangan BLE
Berat paket dibaca dari timbangan digital yang terhubung via Bluetooth Low Energy. Sistem otomatis mendeteksi UUID characteristic BLE yang, sehingga tidak perlu konfigurasi UUID manual.

- Auto-reconnect jika BLE terputus sebanyak 3 kali percobaan.
- Deteksi stability flag dari byte terakhir dari notifikasi BLE.
- Berat di-reset ke 0 otomatis ketika BLE disconnect agar tidak terjadi trigger palsu.

## Integrasi Barcode Scanner via TCP
Scanner barcode terhubung ke sistem melalui TCP server yang berjalan di port 9004. Scanner mengirim string barcode setiap kali scan berhasil.

- Barcode Debounce: barcode yang sama tidak diproses ulang dalam interval waktu tertentu.
- Barcode Expired: Barcode yang tidak diikuti berat stabil dalam batas waktu tertentu (dapat di konfigurasi) akan dianggap expired dan direset.

## Antrian Barcode (FIFO Queue)
Ketika beberapa paket di-scan berturut-turut sebelum transaksi sebelumnya selesai, barcode tambahan **tidak dibuang** melainkan dimasukkan ke antrian FIFO (kapasitas maksimal 10). Setelah satu transaksi selesai dan barang diangkat, barcode berikutnya otomatis ditarik dari antrian.

- Dedup: barcode yang sudah aktif atau sudah ada di antrian tidak ditambahkan dua kali.
- Jika antrian penuh (10), barcode baru di-drop dan sistem mem-publish status `QUEUE_FULL` ke topic `DISPLAY/BARCODE` agar operator tahu.

## Trigger Logic
Transaksi pengukuran hanya dieksekusi ketika tiga kondisi terpenuhi dan bertahan selama `stability_delay` detik:

:::info{title="Kondisi Trigger"}

- Barcode sudah di-scan.
- Berat stabil > 0.02 kg.
- Barang terdeteksi dan bisa diukur (ada di area titik tengah frame).

:::

Ini mencegah pengiriman data saat masih bergerak atau belum ada barcode yang di scan.

:::warning{title="Safety: Tunggu Barang Diangkat"}

Setelah satu transaksi selesai, sistem masuk state **menunggu barang diangkat** (`_awaiting_item_removal`). Trigger berikutnya diblokir sampai berat kembali ≤ 0.02 kg (barang fisik diangkat dari timbangan). Ini mencegah satu paket terukur dua kali. Setelah barang diangkat, barcode berikutnya ditarik dari antrian.

:::

## Kalibrasi Dimensi
Sistem mengoreksi error sistematis pengukuran menggunakan **regresi linear per sisi**. Setiap sisi (panjang/pendek/tinggi) dikoreksi dengan rumus `final = slope × raw + intercept` yang difit ke nilai mentah hasil pengukuran. Operator mengukur beberapa objek referensi yang sudah diketahui ukurannya, dan sistem menghitung parameter regresi optimal dari data tersebut.

### Pengambilan Sampel per Objek
Berbeda dari pengukuran transaksi biasa, kalibrasi mengambil **beberapa sampel manual (N, default 5)** untuk tiap objek referensi. Tiap sampel diambil pada orientasi berbeda (lurus / diagonal kiri / diagonal kanan / dst) dan diperlakukan sebagai satu observasi regresi tersendiri (measured berbeda, actual sama). Ini membuat MAE jujur mencerminkan variasi orientasi dan mencegah fit overfit ke satu posisi.

| Model | Rumus | Catatan |
| --- | --- | --- |
| Linear | `final = slope × raw + intercept` | Model yang dipakai produksi (config). |

:::tip

Gunakan **minimal 2 objek referensi** (`MIN_POINTS`). Direkomendasikan **≥4 objek dengan ukuran bervariasi** — satu kecil (~15cm) dan satu besar (~50cm) — agar slope+intercept ter-fit benar dan model Poly-2 untuk tinggi bisa dihitung. Dengan hanya 1 ukuran, regresi degrade ke rasio sederhana (intercept=0).

:::

- Capture (`/calibrate/measure`) hanya berhasil jika ada objek terdeteksi di **tengah frame** dan bacaan live masih segar (tidak lebih lama dari 1.5 detik). Jika tidak, sistem menolak dengan error 400.
- Sampel maupun data point bisa dihapus untuk diulang tanpa me-restart sesi — gunakan `DELETE /calibrate/measure/{i}` (hapus 1 sampel di buffer berjalan) atau `DELETE /calibrate/point/{i}` (hapus 1 objek referensi).
- Standardisasi long/short (max/min L,W) konsisten dengan pengukuran — jika input L/W tertukar.
- Riwayat semua sesi kalibrasi (parameter ketiga model + MAE) disimpan ke `calibration.json`.
- Setelah model terbaik dipilih (MAE terkecil), terapkan ke produksi via `PUT /config` (`slope_*`/`intercept_*`) — efektif langsung tanpa restart.

## Live Stream & Developer Mode
Frame hasil pemrosesan di-stream via MJPEG HTTP di endpoint /stream. Dashboard dapat menampilkan preview kamera secara realtime.

- Frame di-resize ke 720x540 sebelum stream untuk efisiensi bandwidth.
- FPS stream di-cap 24fps.
- Dev Mode: Aktifkan dengan `DEV_MODE=true` di `.env` untuk mode debugging.

:::info{title="Dev Mode"}

Saat `DEV_MODE=true`, stream menampilkan grid 2x2 — **RAW**, **DEPTH**, **MASKING**, dan **RESULT** — sehingga memudahkan debugging pipeline pengukuran secara visual.

:::

## Runtime Config via HTTP API
Semua parameter tuning dapat diubah saat runtime melalui PUT /config tanpa perlu restart sistem. Perubahan langsung efektif dan disimpan ke config.json untuk persistensi.

:::note

Perubahan config via API bersifat persisten — nilai tersimpan di `config.json` dan tetap berlaku setelah restart.

:::

## Auto-Restart & Recovery Kamera
Jika kamera depth berhenti mengirim frame (stuck), sistem tidak langsung mati. Setelah mencapai batas kegagalan frame berturut-turut, pipeline kamera di-restart otomatis tanpa mematikan program.

- Auto-restart dicoba hingga 3 kali. Jika kamera tetap tidak pulih, baru sistem melaporkan error fatal ke MQTT dan exit.
- Counter restart di-reset setelah kamera benar-benar pulih (frame kembali normal).
- Restart juga bisa dipicu manual via endpoint `POST /restart` (mis. dari dashboard) tanpa restart program.

## Resiliensi MQTT Offline
Jika broker MQTT sedang offline saat sebuah transaksi selesai, hasil pengukuran **tidak hilang**. Payload disimpan sebagai *pending result* dan otomatis dikirim ulang (flush) begitu koneksi MQTT pulih kembali.

- Auto-reconnect ke broker dengan retry tiap 5 detik.
- LWT (Last Will and Testament): jika sistem mati mendadak, broker otomatis mem-publish status `OFFLINE` ke topic `STATUS`.

## Transaction Profiler
Untuk analisis performa, setiap transaksi dapat di-profil: waktu tiap tahap lifecycle (barcode diterima → berat stabil → trigger → publish MQTT) dicatat ke file CSV (`transaction_logs/transactions.csv`).

- Mencatat durasi tahapan kunci: waktu pergerakan barang, hold delay aktual, durasi proses, durasi publish MQTT, dan total durasi.
- Diaktifkan via `PROFILE_ENABLED=True` (default aktif).
- Opsional: rekam video MP4 per transaksi via `PROFILE_VIDEO_ENABLED=True` (lebih berat; output di `transaction_logs/videos/`).

## Error Reporting via MQTT
Error fatal dari modul manapun dikirim ke MQTT topic DWS-SICEPAT/STATUS/ERROR sebelum proses exit, sehingga backend dapat mendeteksi dan menampilkan notifikasi ke operator.

## MQTT Payload

| Topic | Trigger | Isi |
| --- | --- | --- |
| DISPLAY/BARCODE | Tiap scan barcode / barcode di-clear | \{ barcode: str \} (string kosong = cleared) |
| DISPLAY/BARCODE | Antrian penuh | \{ status: "QUEUE\_FULL", barcode, reason \} |
| DISPLAY/MEASUREMENT | Tiap frame (~15fps) | \{ weight, length, width, height, volumetric, divisor \} |
| DATA/RESULT | Saat transaksi selesai | \{ barcode, weight, length, width, height, volumetric, divisor, image\_base64 \} |
| STATUS | Connect / disconnect | \{ status: "ONLINE" \} atau \{ status: "OFFLINE", reason \} |
| STATUS/ERROR | Error fatal | \{ status: "ERROR", failed\_module, error\_message \} |

:::note

Topic dipublish dengan prefix `DWS-SICEPAT/` (mis. `DWS-SICEPAT/DATA/RESULT`). `image_base64` pada `DATA/RESULT` diambil dari kamera RGB sekunder jika tersedia. Saat MQTT reconnect, sistem mem-publish `STATUS/ERROR` kosong (retained) untuk membersihkan error lama.

:::

# Setup

## Prerequisites
- Windows 10/11 64-bit
- Python 3.11
- uv (Astral) - package manager (pip install uv)
- Orbbec SDK DLL tersedia di root project (OrbbecSDK.dll; OrbbecSDKConfig.xml).
- MQTT broker berjalan di localhost: 1883 (atau ws 9001)

## Instalasi

```bash title="Terminal"
# 1. Clone atau extract project
cd dws-sicepat-middleware

# 2. Install dependencies
uv sync

# 3. Install pyorbbecsdk dari wheel lokal
uv add pyorbbecsdk-2.0.15-cp311-cp311-win_amd64.whl

# 4. Buat .env dari template (lihat bagian konfigurasi)
copy .env.example .env

# 5. Jalankan
uv run -m main
```

## Deployment (Build Executable)
Untuk deployment di mesin produksi (tanpa Python environment), build executable Windows menggunakan PyInstaller:

```bash title="Terminal"
uv run pyinstaller dws-middleware.spec --clean
```

Output: `dist/DWS-SICEPAT.exe`. Packaging installer lebih lanjut menggunakan Inno Setup.
